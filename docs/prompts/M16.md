# M16 — Fluxo de "tenho interesse" + mensageria interna (backend)

Leia o CLAUDE.md antes de começar.

## Objetivo

Schema e endpoints para: comprador manifesta interesse num listing → cria thread de conversa → comprador e vendedor trocam mensagens dentro do Códice → vendedor recebe notificação (badge de "não lidas"). Sem revelar dados pessoais de nenhum dos lados. Sem real-time (polling no frontend, M17).

## Contexto

Entities existentes: User, Book, Seller, Listing, ListingPhoto.
Auth: JWT com userId, email, profileType, isAdmin.
Listing tem status ACTIVE (só listings ativos aceitam interesse).
Seller tem relação 1:1 com User.

## Decisões

- Modelo: thread por par (comprador, listing). Um comprador só pode ter 1 thread por listing. Se já existe, retorna a existente.
- Qualquer user autenticado pode manifestar interesse (inclusive INDIVIDUAL_SELLER comprando de outro vendedor). O único impedimento: não pode manifestar interesse no próprio listing.
- Mensagem de texto puro, sem anexos, sem formatação. Max 2000 chars.
- Vendedor vê todas as threads dos seus listings. Comprador vê todas as threads que iniciou.
- Contagem de mensagens não lidas por thread, calculada por participant.
- Sem notificação por email (milestone futuro). Apenas badge visual no frontend (M17).
- Sem bloqueio/report de usuários (milestone futuro).

## O que construir

### 1. Migration V10 — tabelas de mensageria

`V10__create_interest_and_messages.sql`:

**Tabela `interest_threads`:**
- `id` UUID PK default gen_random_uuid()
- `listing_id` UUID NOT NULL REFERENCES listings(id) ON DELETE CASCADE
- `buyer_id` UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE
- `status` VARCHAR(20) NOT NULL DEFAULT 'OPEN' — CHECK IN ('OPEN', 'CLOSED', 'SOLD')
- `last_message_at` TIMESTAMP WITH TIME ZONE — nullable, atualizado a cada mensagem nova
- `created_at` TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
- `updated_at` TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()

Constraints:
- UNIQUE(listing_id, buyer_id) — um comprador, uma thread por listing
- Índice em buyer_id
- Índice em listing_id

**Tabela `messages`:**
- `id` UUID PK default gen_random_uuid()
- `thread_id` UUID NOT NULL REFERENCES interest_threads(id) ON DELETE CASCADE
- `sender_id` UUID NOT NULL REFERENCES users(id) ON DELETE SET NULL
- `content` TEXT NOT NULL CHECK(length(content) > 0 AND length(content) <= 2000)
- `created_at` TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()

Índice:
- Em thread_id, created_at ASC (pra listar mensagens cronologicamente)

**Tabela `thread_read_status`:**
- `thread_id` UUID NOT NULL REFERENCES interest_threads(id) ON DELETE CASCADE
- `user_id` UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE
- `last_read_at` TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
- PRIMARY KEY (thread_id, user_id)

Essa tabela rastreia quando cada participante leu a thread pela última vez. Mensagens com created_at > last_read_at são "não lidas". Simples, sem campo booleano por mensagem.

Aplicar trigger de updated_at em interest_threads.

### 2. Entities JPA

**`br.com.codice.api.interest.InterestThread`:**
- @Entity @Table(name = "interest_threads")
- Relações: @ManyToOne LAZY Listing, @ManyToOne LAZY User (buyer)
- Enum ThreadStatus { OPEN, CLOSED, SOLD }
- Campo lastMessageAt (OffsetDateTime, nullable)
- @OneToMany messages (mappedBy, não precisa carregar eager — queries específicas resolvem)

**`br.com.codice.api.interest.Message`:**
- @Entity @Table(name = "messages")
- Relações: @ManyToOne LAZY InterestThread, @ManyToOne LAZY User (sender)
- Campo content (String)

**`br.com.codice.api.interest.ThreadReadStatus`:**
- @Entity @Table(name = "thread_read_status")
- @IdClass ou @EmbeddedId com (threadId, userId)
- Campo lastReadAt (OffsetDateTime)

### 5. Service

**`br.com.codice.api.interest.InterestService`:**

`createInterest(User buyer, CreateInterestRequest request)`:
- Validar listing existe e está ACTIVE
- Validar buyer não é dono do listing (buscar seller do listing, comparar userId)
- Se thread já existe pra (listing, buyer): retornar a thread existente + adicionar a mensagem
- Se não: criar thread com status OPEN, criar primeira mensagem, setar last_message_at, criar ThreadReadStatus pros dois participantes (buyer e seller-user)
- Retornar ThreadResponse

`getMyThreads(User user, Pageable pageable)`:
- Se user tem seller: buscar threads onde buyerId = userId OU listing.seller.userId = userId
- Se não tem seller: buscar threads onde buyerId = userId
- Ordenar por lastMessageAt DESC (threads com atividade recente primeiro)
- Calcular unreadCount pra cada thread (comparar lastReadAt com mensagens)
- Retornar Page<ThreadResponse>

`getThreadMessages(User user, UUID threadId, Pageable pageable)`:
- Validar user é participante da thread (buyer ou seller-user)
- Buscar mensagens paginadas, ordenadas por created_at ASC
- Marcar como lida: upsert ThreadReadStatus com lastReadAt = now()
- Retornar Page<MessageResponse>

`sendMessage(User user, UUID threadId, SendMessageRequest request)`:
- Validar user é participante da thread
- Validar thread status é OPEN (não CLOSED/SOLD)
- Criar mensagem
- Atualizar thread.lastMessageAt = now()
- Retornar MessageResponse

`getUnreadCount(User user)`:
- Contar threads onde: user é participante E existe mensagem com created_at > lastReadAt do user
- Retornar UnreadCountResponse

### 6. Controller

**`br.com.codice.api.interest.InterestController`:**

- `POST /interests` — autenticado. Cria interest/thread com primeira mensagem. Retorna 201 com ThreadResponse.
- `GET /interests/threads` — autenticado. Minhas threads (como buyer e como seller). Paginado. Retorna Page<ThreadResponse>.
- `GET /interests/threads/{threadId}/messages` — autenticado (só participantes). Marca como lida. Paginado. Retorna Page<MessageResponse>.
- `POST /interests/threads/{threadId}/messages` — autenticado (só participantes, thread OPEN). Retorna 201 com MessageResponse.
- `GET /interests/unread-count` — autenticado. Retorna UnreadCountResponse (pra badge no header).

## Critério de pronto

1. App sobe, V10 aplica sem erro
2. Buyer manifesta interesse → thread criada, mensagem salva, retorna 201
3. Mesmo buyer no mesmo listing → retorna thread existente + nova mensagem
4. Buyer em listing diferente → nova thread
5. Seller do listing vê a thread em GET /interests/threads
6. Buyer vê a thread em GET /interests/threads
7. Ambos veem as mensagens em GET /interests/threads/{id}/messages
8. Seller responde via POST → mensagem criada
9. Unread count do buyer sobe após seller responder
10. Buyer lê mensagens → unread count volta a 0
11. Interesse no próprio listing → 400
12. Listing não-ACTIVE → 400
13. User não-participante tenta ver/mandar mensagem → 403
14. ThreadResponse.counterpartName mostra o nome correto (buyer vê nome do seller, seller vê nome do buyer)
15. ThreadResponse.unreadCount calcula corretamente por participante
16. interestCount aparece no BookDetailResponse/ListingOfferResponse
17. Mensagem vazia → 400
18. Mensagem > 2000 chars → 400

## Pegadinhas conhecidas

- ThreadReadStatus usa chave composta (threadId, userId). JPA com @IdClass exige classe separada com equals/hashCode. Se usar @EmbeddedId, precisa de @Embeddable. Qualquer um funciona, mas não misture.
- "Minhas threads" pra quem é seller E buyer: precisa union das duas queries (threads onde buyerId = eu + threads onde listing.seller.userId = eu). Pode ser query nativa ou dois selects combinados no service. Cuidado pra não duplicar se user mandou interesse pra um listing e também é seller de outro listing na mesma thread (cenário impossível pela constraint, mas pense no SQL).
- lastMessageAt na thread precisa ser atualizado em TODA nova mensagem, não só na criação da thread. Sem isso, a ordenação por atividade recente quebra.
- Marcar como lida (upsert em ThreadReadStatus) deve acontecer ao BUSCAR mensagens, não ao listar threads. O user pode ver a lista de threads sem ter lido as mensagens.
- unreadCount é por participant, não global. Seller pode ter 0 não lidas numa thread enquanto buyer tem 3.
- N+1 trap: ao listar threads, precisa carregar book.title, listing.priceCents, counterpart name. Prefira projeção com JOIN em query nativa a carregar lazy one-by-one.
