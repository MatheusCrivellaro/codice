# M15 — Página do livro + busca full-text com filtros

Leia o CLAUDE.md antes de começar.

## Objetivo

Backend: busca full-text com Postgres tsvector + filtros por área acadêmica, faixa de preço, condição e localização. Frontend: página pública do livro com lista de ofertas, página de busca com filtros laterais, e home redesenhada mostrando curadoria em vez do healthcheck.

Este é o milestone onde o Códice começa a parecer um marketplace navegável.

## Contexto

Backend expõe:
- `GET /books` — paginado, lista books com listings ACTIVE (M10)
- `GET /books/{slug}` — detalhes de um book (M10)
- Entities: Book (com `academic_areas TEXT[]`), Listing (com `condition`, `price_cents`, `status`), Seller (com `city`, `state`)

Extensões Postgres ativas: `pg_trgm` (fuzzy), `tsvector/tsquery` (full-text). Índice GIN em `books.title` com `gin_trgm_ops` já existe (M09).

Frontend tem: React Router, TanStack Query, shadcn/ui, helpers de formatação em `src/lib/format.ts` (formatPrice, formatCondition, etc).

Paleta da marca (CLAUDE.md): Papel #F7F3EC, Tinta #2A2420, Bordô #7A2E2E, Cinza-quente #A89F92.

## O que construir — BACKEND

### 1. Migration V9 — índice full-text

`V9__add_fulltext_search.sql`:

```sql
-- Coluna tsvector materializada pra busca full-text
ALTER TABLE books ADD COLUMN search_vector tsvector;

-- Popula com dados existentes
UPDATE books SET search_vector =
    setweight(to_tsvector('portuguese', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('portuguese', coalesce(authors, '')), 'B') ||
    setweight(to_tsvector('portuguese', coalesce(synopsis, '')), 'C') ||
    setweight(to_tsvector('portuguese', coalesce(publisher, '')), 'D');

-- Índice GIN pra busca rápida
CREATE INDEX idx_books_search_vector ON books USING GIN (search_vector);

-- Trigger pra manter atualizado
CREATE OR REPLACE FUNCTION books_search_vector_update() RETURNS trigger AS $$
BEGIN
    NEW.search_vector :=
        setweight(to_tsvector('portuguese', coalesce(NEW.title, '')), 'A') ||
        setweight(to_tsvector('portuguese', coalesce(NEW.authors, '')), 'B') ||
        setweight(to_tsvector('portuguese', coalesce(NEW.synopsis, '')), 'C') ||
        setweight(to_tsvector('portuguese', coalesce(NEW.publisher, '')), 'D');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_books_search_vector
    BEFORE INSERT OR UPDATE ON books
    FOR EACH ROW
    EXECUTE FUNCTION books_search_vector_update();
```

Pesos: A = título (mais relevante), B = autores, C = sinopse, D = editora.

### 2. Atualizar entity Book

Adicionar campo `searchVector` mapeado como `@Column(columnDefinition = "tsvector", insertable = false, updatable = false)`. Hibernate não precisa gerenciar — o trigger cuida. Mas a coluna precisa existir no mapeamento pro `validate` não reclamar.

### 3. Endpoint de busca

Refatorar `BookController`:

`GET /books/search?q=...&area=...&condition=...&priceMin=...&priceMax=...&state=...&sort=...&page=0&size=20`

Parâmetros (todos opcionais):
- `q` String — query de busca. Se presente, usa full-text; se vazio, retorna tudo.
- `area` String — filtro por área acadêmica (match exato em qualquer elemento do array `academic_areas`)
- `condition` String — filtro por condição mínima (ex: se "BOM", inclui BOM, MUITO_BOM, COMO_NOVO, NOVO). Hierarquia: ACEITAVEL < BOM < MUITO_BOM < COMO_NOVO < NOVO.
- `priceMin` Integer — preço mínimo em centavos
- `priceMax` Integer — preço máximo em centavos
- `state` String — UF do seller (2 chars)
- `sort` String — opções: "relevance" (default quando q presente), "price_asc", "price_desc", "newest" (default quando q ausente)
- `page`, `size` — paginação padrão

A query deve:
- Só retornar books que têm pelo menos 1 listing ACTIVE
- Quando `q` presente, usar `plainto_tsquery('portuguese', :q)` contra `search_vector`. Se tsquery retornar 0 resultados, fallback pra `pg_trgm` similarity no título (fuzzy).
- Aplicar filtros cumulativamente (AND entre todos)
- Retornar `BookSearchResult` com: dados do book + `activeListingsCount`, `lowestPriceCents`, `relevanceScore` (ts_rank quando busca full-text, null caso contrário)

Implementar como query nativa no `BookRepository` (@Query nativeQuery=true) com SQL dinâmico via Specification ou Criteria, OU usando um `BookSearchService` que monta a query nativa com StringBuilder. Recomendação: query nativa com StringBuilder no service — Specifications ficam confusas pra queries com JOIN e agregação.

### 5. Endpoint de áreas acadêmicas disponíveis

`GET /books/academic-areas` — público, retorna lista distinta de áreas acadêmicas que existem em books com listings ACTIVE. Pra popular o filtro no frontend.

Query nativa:
```sql
SELECT DISTINCT unnest(academic_areas) AS area
FROM books b
WHERE EXISTS (SELECT 1 FROM listings l WHERE l.book_id = b.id AND l.status = 'ACTIVE')
ORDER BY area
```

Retorno: `List<String>`.

### 6. Endpoint de UFs com sellers ativos

`GET /sellers/active-states` — público, retorna lista distinta de UFs que têm sellers com listings ACTIVE.

Retorno: `List<String>`.

## Pegadinhas conhecidas

- `search_vector` precisa estar no mapeamento JPA como `insertable = false, updatable = false` senão Hibernate tenta setar null e o validate reclama.
- `plainto_tsquery('portuguese', '')` com string vazia pode dar comportamento inesperado — tratar no service (se q vazio, não aplicar filtro de full-text).
- Filtro de condição por hierarquia: se o user seleciona "BOM", query precisa incluir BOM, MUITO_BOM, COMO_NOVO, NOVO. Implementar como `condition IN (...)` com os valores calculados a partir da seleção.
- Preço nos filtros é em reais no frontend (user digita "40"), em centavos na API (priceMax=4000). Converter na hora de montar a URL.
- `academic_areas` é TEXT[] no Postgres. Filtrar com `@> ARRAY[:area]::text[]` (operador de array contains). Indexado pelo GIN que já existe (M09).
- URL query params devem sobreviver a navegação e refresh. Usar `useSearchParams` do React Router, não state local.
- Imagens de capa do Google Books podem ser HTTP. Converter pra HTTPS no frontend se necessário (ou resolver no backend — já deveria estar resolvido desde M12, mas confere).
