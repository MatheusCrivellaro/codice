# M18 — Deploy do backend no Railway

Leia o CLAUDE.md antes de começar.

## Objetivo

Backend Spring Boot rodando no Railway com Postgres gerenciado, variáveis de ambiente configuradas, deploy automático via push no main, healthcheck acessível publicamente via HTTPS.

Este milestone NÃO envolve código de feature. É configuração de infra e ajustes no backend pra rodar em produção.

## Contexto

Backend em `backend/` — Spring Boot 3, Java 21, Maven, Flyway.
Postgres local via Docker Compose em `infra/`.
Variáveis sensíveis hoje hardcoded no `application.yml`:
- Credenciais do Postgres (codice/codice_dev)
- JWT secret (codice.jwt.secret)
- Admin seed secret (codice.admin.seed-secret)
- Google Books API key (codice.google-books.api-key)

O Railway oferece Postgres gerenciado e deploy de containers. Free trial / Hobby plan (~US$ 5/mês).

## O que construir

### 1. Profile de produção no Spring Boot

Criar `backend/src/main/resources/application-prod.yml`:

```yaml
spring:
  datasource:
    url: ${DATABASE_URL}
    username: ${DATABASE_USERNAME}
    password: ${DATABASE_PASSWORD}

  jpa:
    hibernate:
      ddl-auto: validate
    open-in-view: false
    properties:
      hibernate:
        dialect: org.hibernate.dialect.PostgreSQLDialect

  flyway:
    enabled: true
    locations: classpath:db/migration
    baseline-on-migrate: true

server:
  port: ${PORT:8080}

management:
  endpoints:
    web:
      exposure:
        include: health
  endpoint:
    health:
      show-details: never
      show-components: never

codice:
  jwt:
    secret: ${JWT_SECRET}
    expiration-seconds: 604800
  admin:
    seed-secret: ${ADMIN_SEED_SECRET}
  google-books:
    api-key: ${GOOGLE_BOOKS_API_KEY:}
```

Diferenças do dev:
- Datasource via env vars (Railway injeta)
- PORT via env var (Railway define)
- Health details escondidos (segurança)
- Tudo sensível via env var

### 2. Atualizar CORS pra aceitar domínio de produção

Atualizar `CorsConfig.java` pra ler allowed origins de env var:

```java
// Adicionar ao CorsConfig
@Value("${codice.cors.allowed-origins:http://localhost:5173,http://localhost:5174}")
private String allowedOrigins;
```

E usar `Arrays.asList(allowedOrigins.split(","))` no `config.setAllowedOrigins()`.

Em `application-prod.yml`, adicionar:
```yaml
codice:
  cors:
    allowed-origins: ${CORS_ALLOWED_ORIGINS}
```

### 3. Dockerfile

Criar `backend/Dockerfile`:

```dockerfile
FROM eclipse-temurin:21-jdk-alpine AS build
WORKDIR /app
COPY .mvn/ .mvn/
COPY mvnw pom.xml ./
RUN chmod +x mvnw && ./mvnw dependency:go-offline -B
COPY src/ src/
RUN ./mvnw package -DskipTests -B

FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENV SPRING_PROFILES_ACTIVE=prod
ENTRYPOINT ["java", "-jar", "app.jar"]
```

Multi-stage: build com JDK, runtime com JRE. Imagem final ~200MB.

### 5. Testar build local do Docker

Antes de subir pro Railway, confirmar que builda e roda local:

```bash
cd backend
docker build -t codice-api .
docker run --rm -p 8080:8080 \
  -e DATABASE_URL=jdbc:postgresql://host.docker.internal:5432/codice \
  -e DATABASE_USERNAME=codice \
  -e DATABASE_PASSWORD=codice_dev \
  -e JWT_SECRET=$(openssl rand -base64 48) \
  -e ADMIN_SEED_SECRET=test-secret \
  -e SPRING_PROFILES_ACTIVE=prod \
  codice-api
```

Deve subir e /actuator/health responder (sem detalhes, porque prod esconde).

## Deploy contínuo

Todo push na branch main redeploya automaticamente.

## Postgres — extensões

Railway Postgres não habilita extensões automaticamente. Após o primeiro deploy:

1. Conectar no Postgres do Railway (credenciais no Dashboard → Variables)
2. Executar:
```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE EXTENSION IF NOT EXISTS vector;
```

O Flyway cuida do resto (migrations aplicam automaticamente).

## Troubleshooting

- **App não sobe:** Railway Logs no dashboard. Geralmente é variável faltando.
- **Flyway falha:** extensão não habilitada. Ver seção acima.
- **CORS bloqueado:** CORS_ALLOWED_ORIGINS não inclui o domínio do frontend.
- **Cold start lento (~15-30s):** normal no Hobby plan. JVM demora pra aquecer.
```

## Pegadinhas conhecidas

- Railway injeta DATABASE_URL no formato `postgresql://user:pass@host:port/db`, mas Spring Boot espera `jdbc:postgresql://...`. Se Railway usar o formato sem `jdbc:`, você precisa de um adapter ou usar a variável `PGHOST`, `PGPORT`, etc. separadamente. Verificar qual formato Railway usa nas Reference Variables e ajustar o application-prod.yml.
- Railway Postgres pode não ter as extensões pgcrypto, pg_trgm, vector habilitadas. Se Flyway falhar na primeira migration com "function gen_random_uuid() does not exist", são as extensões. Documentar no deploy guide.
- PORT no Railway pode não ser 8080. Usar `${PORT:8080}` no yml.
- Health details devem ser escondidos em prod (show-details: never) — não expor estado do DB publicamente.
- Dockerfile precisa do `chmod +x mvnw` senão falha no Alpine.

Mostre o plano antes de começar.
