# M23 — Layout refinado e responsividade

Leia o CLAUDE.md antes de começar. Referência visual: Cap. 6 do Documento Mestre em `docs/Codice_Documento_Mestre.pdf` (páginas 16-18).

## Objetivo

Refinar o layout de todas as páginas pra refletir as diretrizes do Doc Mestre: largura máxima 1280px, blocos de leitura em 680px, grid de livros 4 colunas desktop / 2 mobile, espaçamento generoso. Garantir que toda página funciona bem de 375px (iPhone SE) a 1440px (desktop grande). Refinar o AppShell (header/navegação) pra parecer navegação de livraria, não de SaaS.

Não é redesign — é polimento do que já existe usando os tokens do M22.

## Contexto

Design tokens do M22 já aplicados: fontes (Fraunces, Source Serif 4, Inter), paleta (papel, tinta, bordô, etc.), componentes shadcn estilizados.

Páginas existentes:
- Home (hero + curadoria + áreas acadêmicas)
- SearchPage (filtros laterais + grid de resultados)
- BookPage (detalhes do livro + ofertas)
- LoginPage, RegisterPage
- CreateListingWizard (4 passos)
- MyListings
- SellerProfileForm, SellerDashboard
- ConversationsPage, ConversationDetailPage
- Admin: ModerationQueue, AllListings
- PerfilPage
- NotFoundPage

## O que construir

Refatorar `src/components/layout/AppShell.tsx`:

**Header desktop (>= 768px):**
- Fundo: papel #F7F3EC, borda inferior sutil (1px cinza-borda)

Adicionar shadcn Sheet se não existir:


### 2. Container de conteúdo

Criar ou confirmar existência de `container-codice` (definido no M22). Aplicar em TODAS as páginas como wrapper principal:

```tsx
<main className="container-codice py-8">
  {/* conteúdo da página */}
</main>
```

Garantir que nenhuma página ultrapasse 1280px de conteúdo. O fundo da página (papel) vai edge-to-edge, mas o conteúdo é contido.

### 14. Breakpoints

Usar os breakpoints padrão do Tailwind:
- sm: 640px (ajustes pontuais)
- md: 768px (switch pra layout mobile: menu hambúrguer, filtros em drawer, grids 2 col)
- lg: 1024px (grid 3→4 colunas, sidebar visível)
- xl: 1280px (max-width do conteúdo)

Testar em: 375px (iPhone SE), 390px (iPhone 14), 768px (iPad), 1024px (iPad landscape), 1280px+ (desktop).

## O que NÃO fazer

- Não implementar dark mode (M24)
- Não adicionar animações complexas (transições, parallax, scroll animations)
- Não alterar lógica de negócio
- Não adicionar páginas novas
- Não mexer em endpoints do backend
- Não trocar componentes shadcn por biblioteca externa
- Não otimizar imagens (lazy loading de imagens é ok se simples, mas não prioritário)

## Critério de pronto

1. Header: logo Fraunces, busca centralizada (desktop), hambúrguer funcional (mobile)
2. Footer presente em todas as páginas
3. Home: hero com busca, grid de livros 4col/2col, áreas clicáveis
4. Busca: sidebar de filtros (desktop), drawer (mobile), grid responsivo
5. Página do livro: layout 2 colunas (desktop), empilhado (mobile), ofertas com fotos
6. Forms: centrados, inputs confortáveis, botões full-width
7. Chat: bolhas com visual de mensageiro, max-width de leitura
8. Admin: tabela navegável, ações visíveis
9. 404: centrado, microcopy do Códice
10. Nenhum conteúdo ultrapassando 1280px
11. Nenhum texto cortado ou overflow horizontal em 375px
12. Scrollbar horizontal nunca aparece em nenhuma viewport
13. Touch targets >= 44px no mobile (botões, links, checkboxes)
14. Espaçamento generoso: nada apertado, tudo respira
15. Zero erros no console

## Pegadinhas conhecidas

- Sticky sidebar pode conflitar com o footer se a página for curta. Usar `sticky top-[80px]` com `self-start` e testar com pouca e muita conteúdo.
- Menu hambúrguer: se usar shadcn Sheet, garantir que fecha ao clicar num link de navegação (não fica aberto após navegar).
- Grid 4 colunas com gap 24px dentro de 1280px: cada card fica ~290px. Se as capas forem menores, ficam esticadas. Usar `object-contain` ou `object-cover` na imagem.
- Line-clamp precisa de `-webkit-line-clamp` com `display: -webkit-box`. Tailwind tem `line-clamp-2` mas verificar se funciona no build de produção.
- Footer fixo no fundo da página (não da viewport): usar `min-h-screen` com flex column no layout, footer com `mt-auto`.
- Focus states: garantir que todo elemento interativo tem focus ring visível (acessibilidade). O ring bordô do M22 deve cobrir.
