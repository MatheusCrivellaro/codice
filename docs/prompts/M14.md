# M14 — Wizard de cadastro de anúncio (frontend)

Leia o CLAUDE.md antes de começar.

## Objetivo

Wizard de 4 passos no frontend onde o vendedor cadastra um anúncio: busca por ISBN → confirma/preenche dados do livro → preço, condição e fotos → revisão e envio. Consome os endpoints do M13 (POST /listings, POST /sellers/profile, GET /lookup/isbn, GET /books/search/fuzzy).

Sem upload real de fotos (URLs placeholder por enquanto — M19 resolve). Sem sugestão de preço.

## Contexto

Backend expõe:
- `GET /lookup/isbn/{isbn}` → BookLookupResponse (M12)
- `GET /books/search/fuzzy?title=...` → lista de BookFuzzyMatch (M13)
- `POST /sellers/profile` → cria perfil de seller para BOOKSTORE (M13)
- `GET /sellers/me` → perfil do seller logado, ou 404 (M13)
- `POST /listings` → cria listing (M13)
- `GET /listings/mine` → listings do seller logado (M13)

Frontend tem: AuthContext com user (inclui profileType), apiFetch em src/api/client.ts, React Hook Form + Zod instalados, shadcn/ui com form/input/button/card/radio-group/dialog/sonner/textarea/badge/tabs/table.

## Decisões

- Wizard em 4 passos, SEM persistência de rascunho (se abandonar, perde tudo — aceitável no MVP).
- BOOKSTORE sem perfil de seller: ao acessar /vender, intercepta e mostra formulário de perfil antes do wizard.
- INDIVIDUAL_SELLER: vai direto pro wizard (seller auto-provisionado pelo backend no POST /listings).
- BUYER: ao acessar /vender, mostra mensagem "Compradores não podem anunciar" e link pra home.
- Fotos: campo de URL de imagem (input text), mínimo 1. Cada foto tem position (ordem) e photoType (select). O upload real vem no M19.

## O que construir

- `createSellerProfile(data: CreateSellerProfileRequest): Promise<SellerProfileResponse>` — POST /sellers/profile

Usar `apiFetch` de `src/api/client.ts`.

### 2. Roteamento

### 3. SellPage — wrapper de decisão

`src/pages/sell/SellPage.tsx`:

Lógica:
1. Se user.profileType === 'BUYER' → mostrar Card com mensagem "Compradores não podem anunciar livros" + botão pra voltar à home
2. Se user.profileType === 'BOOKSTORE':
   a. Fazer GET /sellers/me
   b. Se 404 → mostrar formulário de perfil de seller (inline ou sub-componente)
   c. Se tem perfil → mostrar painel do vendedor (link pra criar anúncio + link pra meus anúncios)
3. Se user.profileType === 'INDIVIDUAL_SELLER' → mostrar painel do vendedor direto (seller é auto-provisionado no backend)

### 6. Wizard de criação de anúncio — 4 passos

`src/pages/sell/CreateListingWizard.tsx`:

Estado local com `useState` gerenciando o passo atual (1-4) e os dados acumulados entre passos. NÃO usar React Hook Form pra o wizard inteiro — usar form individual por passo com dados consolidados num state pai.

**Passo 1 — Identificar o livro:**

Duas opções apresentadas como tabs ou radio:

a) "Tenho o ISBN": input de ISBN, botão "Buscar". Ao buscar:
   - Chama GET /lookup/isbn/{isbn}
   - Se encontrou: mostra preview do livro (título, autores, editora, ano, capa se houver) com botão "Confirmar este livro" e botão "Não é este, preencher manualmente"
   - Se 404: toast "ISBN não encontrado" e mostra opção de preencher manualmente
   - Se 400: toast "ISBN inválido"

b) "Não tenho ISBN / preencher manualmente": formulário com campos do ManualBookData:
   - Título (required)
   - Autores (required)
   - Editora
   - Ano de publicação
   - Edição
   - Idioma (default pt-BR)
   - Tradutor
   - Áreas acadêmicas (input de tags: digita e aperta Enter pra adicionar, X pra remover — pode ser implementação simples com input + array no state)
   - Sinopse (textarea)

   Ao digitar título com 4+ caracteres, debounce de 500ms chama GET /books/search/fuzzy?title=... e mostra sugestões abaixo do campo: "Este livro já está no acervo?" com lista clicável. Se o vendedor clicar numa sugestão, preenche os campos e avança com bookId existente.

Dados salvos no state do wizard: `{ isbn?, manualBookData?, lookupResult? }`

**Passo 2 — Detalhes do anúncio:**

- Preço em reais: input numérico, com máscara "R$ " no início. Internamente converter pra centavos (ex: usuário digita "35,00" → priceCents = 3500). Mínimo R$ 1,00.
- Estado de conservação: radio group com os 5 valores, cada um com label traduzida e descrição curta:
  - "Novo" — "Nunca lido, sem marcas de uso"
  - "Como novo" — "Lido com cuidado, sem marcas visíveis"
  - "Muito bom" — "Pequenos sinais de uso, sem grifos ou anotações"
  - "Bom" — "Sinais normais de uso, pode ter leves marcas"
  - "Aceitável" — "Desgaste visível, pode ter grifos ou anotações"
- Observações sobre o estado (conditionNotes): textarea opcional, placeholder "Descreva marcas, grifos ou detalhes relevantes sobre o estado do livro"
- Descrição do anúncio: textarea opcional, placeholder "Informações adicionais sobre este exemplar"

**Passo 3 — Fotos:**

Lista de inputs de URL de foto (MVP sem upload real). Cada foto tem:
- URL (input text, required)
- Tipo (select): Capa frontal, Lombada/contracapa, Detalhe interno, Defeito, Folha de rosto, Outro
- Position: definido pela ordem na lista (0, 1, 2...)
- Botão de remover foto
- Botão "Adicionar foto" (máximo 10)

Mínimo 1 foto pra avançar. Validação ao tentar ir pro passo 4.

Texto de orientação acima da lista: "Adicione pelo menos uma foto do livro. Fotos nítidas com boa iluminação ajudam a vender mais rápido."

**Passo 4 — Revisão e envio:**

Card de resumo mostrando:
- Dados do livro (título, autores, editora, ano) — se veio do ISBN, mostrar a capa
- Preço formatado
- Estado de conservação (traduzido)
- Observações e descrição (se preenchidos)
- Thumbnails das fotos (se URLs forem imagens válidas, mostrar; senão, mostrar URL como texto)

Botão "Publicar anúncio" (primário, full width). Texto pequeno abaixo: "Seu anúncio será revisado antes de aparecer no acervo."

Ao submeter:
- Montar o `CreateListingRequest` a partir do state acumulado
- POST /listings
- Se 201: toast "Anúncio enviado para revisão" + redireciona pra /vender/anuncios
- Se 400: toast com mensagem de erro do backend
- Se 403: toast "Você não pode anunciar livros"
- Se 500: toast "Erro inesperado. Tente novamente."

Botão "Voltar" em cada passo retorna ao passo anterior sem perder dados.

## Pegadinhas conhecidas

- Input de preço em reais → conversão pra centavos. Se o user digitar "35,50", priceCents = 3550. Cuidado com ponto flutuante: multiplicar por 100 e arredondar (Math.round), não truncar.
- Debounce no fuzzy search: sem debounce, cada keystroke dispara request. Usar setTimeout com 500ms e limpar no cleanup.
- State do wizard entre passos: se usar React Hook Form individual por passo, os dados do passo anterior precisam estar no state pai (useState no CreateListingWizard), não no form — senão perde ao desmontar.
- O endpoint POST /listings espera `condition` como string do enum (ex: "COMO_NOVO"), não como label traduzida.
- O endpoint POST /listings espera `photoType` como string do enum (ex: "COVER_FRONT"), não como label.
- Se o user clicar numa sugestão fuzzy no passo 1, você NÃO precisa passar bookId no request — o backend resolve. Mas pode preencher os campos de manualBookData com os dados do book sugerido pra o user conferir.
