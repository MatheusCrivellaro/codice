# M19 — Upload de fotos com Cloudflare R2

Leia o CLAUDE.md antes de começar.

## Objetivo

Substituir as URLs placeholder de fotos por upload real. Backend gera URL pré-assinada (presigned URL), frontend faz PUT direto no R2, URL pública volta pro listing. Atualizar o wizard de cadastro de anúncio (M14) pra usar upload em vez de input de URL.

## Contexto

Cloudflare R2: storage S3-compatible com free tier generoso (10GB storage, 10M requests classe A/mês, 0 egress). O fundador já tem conta Cloudflare.

Backend: endpoint POST /listings (M13) recebe fotos como `List<ListingPhotoInput>` com campo `url`. Vamos manter esse contrato — o frontend faz upload, recebe URL pública, passa pro backend como antes.

Frontend: wizard de anúncio (M14) tem passo 3 "Fotos" com inputs de URL. Vamos substituir por drag-and-drop / file picker real.

## Decisões

- Upload via presigned PUT URL — frontend faz upload direto pro R2 sem passar pelo backend. Backend só gera a URL assinada.
- Validação no frontend: máximo 5MB por foto, formatos jpg/jpeg/png/webp, máximo 10 fotos.
- Sem redimensionamento server-side no MVP. Frontend pode redimensionar client-side se imagem for muito grande (nice to have, não obrigatório).
- Sem watermark.
- URLs públicas servidas via domínio público do R2 (ou custom domain se configurado).
- Organização no bucket: `listings/{userId}/{timestamp}-{random}.{ext}`

## O que construir — BACKEND

### 1. Dependência S3

Adicionar ao `pom.xml`:
```xml
<dependency>
    <groupId>software.amazon.awssdk</groupId>
    <artifactId>s3</artifactId>
    <version>2.25.60</version>
</dependency>
```

Versão pode variar — usar a estável mais recente da série 2.x.

### 2. Configuração

Em `application.yml` (dev):
```yaml
codice:
  storage:
    endpoint: ${R2_ENDPOINT}
    access-key: ${R2_ACCESS_KEY}
    secret-key: ${R2_SECRET_KEY}
    bucket: ${R2_BUCKET:codice-uploads}
    public-url: ${R2_PUBLIC_URL}
```

Em `application-prod.yml`, mesma seção com env vars.

Explicação das variáveis:
- `R2_ENDPOINT`: endpoint S3 do R2, formato `https://<account-id>.r2.cloudflarestorage.com`
- `R2_ACCESS_KEY`, `R2_SECRET_KEY`: credenciais S3 do R2 (gerar no Cloudflare Dashboard → R2 → Manage R2 API Tokens)
- `R2_BUCKET`: nome do bucket
- `R2_PUBLIC_URL`: URL base pra acesso público (formato `https://pub-<hash>.r2.dev` ou custom domain)

### 3. Serviço de storage

`br.com.codice.api.storage.StorageService`:

Configurar `S3Client` como bean:
```java
S3Client.builder()
    .endpointOverride(URI.create(endpoint))
    .credentialsProvider(StaticCredentialsProvider.create(
        AwsBasicCredentials.create(accessKey, secretKey)))
    .region(Region.of("auto"))
    .build()
```

E `S3Presigner` como bean (separado do S3Client):
```java
S3Presigner.builder()
    .endpointOverride(URI.create(endpoint))
    .credentialsProvider(...)
    .region(Region.of("auto"))
    .build()
```

Métodos:

`generatePresignedUploadUrl(UUID userId, String filename)`:
- Gerar key: `listings/{userId}/{timestamp}-{randomUUID}.{extensão}`
- Extrair extensão do filename, validar contra whitelist (jpg, jpeg, png, webp)
- Gerar presigned PUT URL com expiração de 15 minutos
- Retornar DTO com `uploadUrl` (presigned URL pra PUT) e `publicUrl` (URL pública final do arquivo)

`deleteFile(String key)`:
- Deleta arquivo do R2. Pra uso futuro quando vendedor remover foto.

### 4. DTOs

`PresignedUploadRequest` (record):
- `filename` String, not blank (ex: "foto-capa.jpg")
- `contentType` String, not blank (ex: "image/jpeg")

`PresignedUploadResponse` (record):
- `uploadUrl` String — URL presigned pra PUT (o frontend faz PUT com o arquivo aqui)
- `publicUrl` String — URL pública final (o frontend guarda essa pra mandar pro POST /listings)
- `key` String — key no bucket (pra referência/deleção futura)

### 5. Controller

`br.com.codice.api.storage.UploadController`:

`POST /uploads/presign` — autenticado. Recebe `PresignedUploadRequest`, retorna `PresignedUploadResponse`.

Pode gerar múltiplas de uma vez:

`POST /uploads/presign-batch` — autenticado. Recebe `List<PresignedUploadRequest>` (max 10), retorna `List<PresignedUploadResponse>`.

Adicionar rotas ao SecurityConfig como autenticadas.

### 6. Validação de content type

No controller, validar que `contentType` é um de: `image/jpeg`, `image/png`, `image/webp`. Retornar 400 se não for.

## O que NÃO fazer

- Não implementar redimensionamento server-side
- Não implementar watermark
- Não implementar CDN custom (R2 public URL basta)
- Não implementar crop/edit de imagem no frontend
- Não implementar drag-to-reorder se for complexo demais (botões ↑↓ servem)
- Não validar dimensões mínimas da imagem (só tamanho em bytes e formato)
- Não implementar deleção de fotos do R2 quando listing é removido (cleanup futuro)

## Pegadinhas conhecidas

- CORS no R2: o bucket precisa de configuração CORS pra aceitar PUT do browser. No Cloudflare Dashboard → R2 → bucket → Settings → CORS, adicionar:
  ```json
  [
    {
      "AllowedOrigins": ["http://localhost:5173", "https://seudominio.com.br"],
      "AllowedMethods": ["PUT", "GET"],
      "AllowedHeaders": ["Content-Type"],
      "MaxAgeSeconds": 3600
    }
  ]
  ```
  Documentar no deploy guide.

- Presigned URL do R2 pode ter formato ligeiramente diferente do S3 padrão. AWS SDK v2 funciona com R2, mas a região deve ser "auto" (não us-east-1).

- `URL.createObjectURL(file)` cria URL temporária pra preview. Precisa chamar `URL.revokeObjectURL()` no cleanup (useEffect return) pra evitar leak de memória.

- Upload paralelo: se o user adicionar 5 fotos de uma vez, todas devem fazer upload em paralelo (Promise.all), não sequencial. Mas limitar concorrência se necessário (3 paralelos max).

- O presigned URL tem validade de 15 minutos. Se o user demorar mais que isso entre selecionar a foto e submeter o listing, o upload pode falhar. Cenário improvável no uso normal, mas se acontecer, gerar nova URL e retry.

- Frontend não deve enviar header Authorization no PUT pro R2 — a presigned URL já contém autenticação. Se enviar Authorization, R2 rejeita.
