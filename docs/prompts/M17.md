# M17 — Frontend do fluxo de interesse e mensageria

Leia o CLAUDE.md antes de começar.

## Objetivo

Botão "Tenho interesse" funcional na página do livro, tela de conversas com lista de threads e chat por thread, badge de não lidas no header, polling de 15s pra novas mensagens. Comprador e vendedor conversam inteiramente dentro do Códice.

## Contexto

Backend expõe (M16):
- `POST /interests` — cria thread + primeira mensagem. Body: { listingId, message }
- `GET /interests/threads?page=0&size=20` — threads do user (como buyer e seller)
- `GET /interests/threads/{threadId}/messages?page=0&size=50` — mensagens, marca como lida
- `POST /interests/threads/{threadId}/messages` — envia mensagem. Body: { content }
- `GET /interests/unread-count` — { totalUnread }

ThreadResponse inclui: id, listingId, bookTitle, bookSlug, bookCoverUrl, listingPriceCents, listingPriceFormatted, listingCondition, counterpartName, counterpartType, status, unreadCount, lastMessagePreview, lastMessageAt, createdAt.

MessageResponse inclui: id, senderId, senderName, isMine, content, createdAt.

Frontend tem: AuthContext, AppShell, BookPage com botão "Tenho interesse" desabilitado (M15), React Router, TanStack Query, shadcn/ui, formatadores em src/lib/format.ts.

## Decisões

- Polling de 15s pra lista de threads e contagem de não lidas (refetchInterval do TanStack Query). Sem WebSocket.
- Polling de 5s quando uma thread está aberta (mensagens da conversa ativa).
- Mensagens novas aparecem no final da lista, scroll automático pro fundo quando chega mensagem nova.
- Sem indicador de "digitando", sem confirmação de leitura visível pro outro participant, sem envio de fotos.
- Chat simples estilo WhatsApp Web: minhas mensagens à direita (com fundo leve), mensagens do outro à esquerda.

### 10. Polling e invalidação

Configuração do TanStack Query:
- `['threads']` — refetchInterval: 15000 (15s), enabled quando autenticado
- `['thread-messages', threadId]` — refetchInterval: 5000 (5s), enabled quando thread aberta
- `['unread-count']` — refetchInterval: 15000 (15s), enabled quando autenticado

Após enviar mensagem (mutation onSuccess):
- Invalidar `['thread-messages', threadId]`
- Invalidar `['threads']` (pra atualizar lastMessagePreview na lista)
- NÃO invalidar unread-count (minhas próprias mensagens não criam unread pra mim)

Após abrir uma conversa (useEffect no mount):
- Invalidar `['unread-count']` (backend marcou como lida no GET messages)

### 11. Scroll automático

No ConversationDetailPage:
- useRef no container de mensagens
- useEffect que scrolla pro fundo quando: componente monta, mensagens mudam (data do useQuery muda)
- Scroll suave (behavior: 'smooth') pra mensagens novas, instantâneo no mount inicial
- Se o user scrollou pra cima manualmente (lendo histórico), NÃO forçar scroll pro fundo — só scrollar se já estava no fundo. Detectar com: scrollHeight - scrollTop - clientHeight < 100

## Pegadinhas conhecidas

- Scroll automático conflitando com leitura de histórico: se o user scrollou pra cima e chega mensagem nova por polling, NÃO roubar o scroll. Só scrollar se já estava no fundo.
- Enter no textarea: por padrão, Enter cria nova linha. Precisa interceptar onKeyDown, checar e.key === 'Enter' && !e.shiftKey, chamar e.preventDefault() e enviar.
- Polling de 5s nas mensagens pode causar flash visual se o componente re-renderizar inteiro. TanStack Query com `structuralSharing: true` (default) evita isso — mas se o container de scroll resetar posição no re-render, é bug de key/ref.
- Badge de não lidas: invalidar ['unread-count'] no onSuccess da query de mensagens (não da mutation de envio). O GET /messages é o que marca como lida no backend.
- Se user não está autenticado, os hooks de polling devem ter `enabled: false` pra não disparar 401 a cada 15s.
- Dialog de "Tenho interesse": se o user fechar e reabrir, o textarea deve estar limpo (não manter estado da tentativa anterior).
- Auto-resize do textarea: setar height pra 'auto' e depois pra scrollHeight no onChange. Limitar maxHeight a 4 linhas (~6rem).
- Ordering de mensagens: backend retorna ASC (mais antigas primeiro). Frontend renderiza na mesma ordem, com scroll posicionado no fundo.
