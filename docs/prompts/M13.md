# M13 — Cadastro de anúncio (backend)

Leia o CLAUDE.md antes de começar.

## Objetivo

Endpoint `POST /listings` que aceita ISBN ou dados manuais, encontra/cria o Book, cria o Listing em PENDING_REVIEW vinculado ao seller autenticado, e associa URLs de fotos. Inclui endpoint de setup de perfil de seller e auto-provision para INDIVIDUAL_SELLER.

Sem frontend. Sem upload real de fotos. Sem sugestão de preço.

## Contexto

Entities existentes: `User`, `Book`, `Seller`, `Listing`, `ListingPhoto` (M10).
Lookup de ISBN: `BookLookupService` em `br.com.codice.api.lookup` (M12).
Slug service: `SlugService` em `br.com.codice.api.common` (M10).
Auth: JWT com claims `email`, `profileType`, `isAdmin`. User autenticado disponível no SecurityContext.

Schema (M09):
- `sellers` tem FK pra `users` (1:1 opcional), campos `public_name`, `slug`, `seller_type`, `description`, `city`, `state`, `neighborhood`
- `listings` tem FK pra `sellers` e `books`, nasce `PENDING_REVIEW`
- `listing_photos` com `url`, `position`, `photo_type`
- `books` com `slug` NOT NULL UNIQUE, `isbn` nullable com unique parcial

User `profileType`: BUYER, BOOKSTORE, INDIVIDUAL_SELLER.

## Decisões

- **INDIVIDUAL_SELLER:** seller criado automaticamente no primeiro anúncio. `public_name = user.name`, `slug = slugify(user.name)`, `seller_type = INDIVIDUAL`. Sem exigir nada extra.
- **BOOKSTORE:** precisa completar perfil de seller ANTES de anunciar. Campos obrigatórios: `public_name`, `description` (min 20 chars), `city`, `state`. Se BOOKSTORE tentar criar listing sem ter seller → 400: "Sebos precisam completar o perfil da loja antes de anunciar. Acesse /vender/perfil."
- **BUYER:** não pode criar listing → 403: "Compradores não podem anunciar livros. Altere seu perfil para vendedor."
- Listing nasce PENDING_REVIEW.
- Preço em centavos (`price_cents`), mínimo 100 (R$ 1,00).
- Fotos como URLs por enquanto. Mínimo 1 foto obrigatória.
- Auto-provision de seller e criação de listing devem acontecer na mesma transação.

## O que construir

### 1. Endpoint de setup de perfil de seller

`br.com.codice.api.seller.SellerController`:

`POST /sellers/profile` — autenticado, qualquer profileType exceto BUYER.

Request DTO (`CreateSellerProfileRequest`, record):
- `publicName` String, not blank, max 150
- `description` String, not blank, min 20, max 2000
- `city` String, not blank, max 120
- `state` String, not blank, exatamente 2 chars (UF)
- `neighborhood` String, nullable, max 120

Response DTO (`SellerProfileResponse`, record):
- `id` UUID
- `publicName`, `slug`, `sellerType`, `description`, `city`, `state`, `neighborhood` (Strings)
- `bannerImageUrl`, `avatarImageUrl` (nullable)
- `createdAt` OffsetDateTime

Comportamento:
- User já tem seller → 409: "Perfil de vendedor já existe."
- Gerar slug via `SlugService.ensureUniqueSellerSlug(slugify(publicName))`
- `sellerType` derivado do `profileType`: BOOKSTORE → BOOKSTORE, INDIVIDUAL_SELLER → INDIVIDUAL
- Retorna 201

`GET /sellers/me` — autenticado, retorna perfil de seller do user logado, ou 404 se não tiver.

### 2. Serviço de criação de listing

`br.com.codice.api.listing.ListingService`, método `createListing(User user, CreateListingRequest request)`:

Fluxo:
1. Verificar profileType:
   - BUYER → throw 403
   - BOOKSTORE → verificar se tem seller, senão throw 400 com mensagem
   - INDIVIDUAL_SELLER → verificar se tem seller, se não criar automaticamente

2. Resolver o book:
   - Se `request.isbn` presente e não-branco:
     a. Normalizar ISBN
     b. Buscar `Book` existente por ISBN no banco
     c. Se não existir, fazer lookup via `BookLookupService`
     d. Se lookup achou, criar Book com dados do lookup + slug gerado
     e. Se lookup não achou E dados manuais vieram no request, criar Book com dados manuais
     f. Se lookup não achou E sem dados manuais → 400: "ISBN não encontrado. Preencha os dados manualmente."
   - Se `request.isbn` ausente:
     a. Dados manuais obrigatórios (título + autores no mínimo)
     b. Buscar book existente por fuzzy match (título + autores via pg_trgm, similarity > 0.6)
     c. Se match forte (> 0.8), reusar book existente
     d. Se match fraco (0.6-0.8), criar novo mas logar warning (possível duplicata)
     e. Se sem match, criar novo

3. Criar listing com status PENDING_REVIEW, publishedAt null

4. Criar fotos (iterar request.photos, criar ListingPhoto pra cada)

5. Retornar ListingResponse

### 3. DTOs

**`CreateListingRequest`** (record):
- `isbn` String, nullable
- `manualBookData` ManualBookData, nullable
- `priceCents` Integer, not null, min 100
- `condition` String, not null — validar contra enum (NOVO, COMO_NOVO, MUITO_BOM, BOM, ACEITAVEL)
- `conditionNotes` String, nullable, max 1000
- `description` String, nullable, max 2000
- `photos` List<ListingPhotoInput>, not null, min size 1, max size 10

Validação custom: isbn e manualBookData não podem ser ambos ausentes. Se isbn presente, manualBookData é opcional (fallback). Se isbn ausente, manualBookData obrigatório com pelo menos title e authors.

**`ManualBookData`** (record):
- `title` String, not blank, max 300
- `authors` String, not blank, max 500
- `publisher` String, nullable, max 200
- `publicationYear` Integer, nullable
- `edition` String, nullable, max 50
- `language` String, default "pt-BR"
- `translator` String, nullable, max 200
- `academicAreas` List<String>, nullable
- `synopsis` String, nullable, max 5000

**`ListingPhotoInput`** (record):
- `url` String, not blank, max 500
- `position` Integer, not null, min 0
- `photoType` String, not null — validar contra enum

**`ListingResponse`** (record):
- `id` UUID
- `bookId` UUID, `bookTitle` String, `bookAuthors` String, `bookSlug` String
- `sellerId` UUID, `sellerName` String
- `priceCents` Integer, `priceFormatted` String (ex: "R$ 45,00")
- `condition` String, `conditionNotes` String (nullable)
- `description` String (nullable)
- `status` String
- `photos` List<ListingPhotoResponse>
- `createdAt` OffsetDateTime

**`ListingPhotoResponse`** (record):
- `id` UUID, `url` String, `position` Integer, `photoType` String

### 4. Controller

`br.com.codice.api.listing.ListingController`:
- `POST /listings` — autenticado, @Valid, chama service, retorna 201
- `GET /listings/mine` — autenticado, paginado. Se user não tem seller, retorna lista vazia.

### 5. Busca fuzzy de books

`GET /books/search/fuzzy?title=...&author=...` — público, até 5 resultados com similarity score via pg_trgm.

Query nativa no BookRepository:
```sql
SELECT *, similarity(title, :title) AS score
FROM books
WHERE similarity(title, :title) > 0.3
ORDER BY score DESC
LIMIT 5
```

DTO: `BookFuzzyMatch(UUID id, String slug, String title, String authors, double score)`.

### 6. Formatação de preço

`br.com.codice.api.common.PriceFormatter`:
- Método estático `format(int cents)` → "R$ 45,00"
- Forçar locale pt-BR explicitamente (não depender do locale do servidor)

### 7. Auto-provision de seller

Método privado no ListingService, `autoProvisionSeller(User user)`:
- `publicName = user.getName()`
- `slug = slugService.ensureUniqueSellerSlug(slugify(user.getName()))`
- `sellerType = INDIVIDUAL`
- Demais campos nullable ficam null
- Salvar e retornar
- Deve estar na MESMA transação que a criação do listing

## Critério de pronto

1. App sobe sem erro
2. INDIVIDUAL_SELLER cria listing sem ter seller → seller auto-provisionado, listing em PENDING_REVIEW
3. Conferir no banco que seller foi criado com dados do user
4. BOOKSTORE sem seller tenta criar listing → 400
5. BOOKSTORE cria perfil via POST /sellers/profile → 201
6. BOOKSTORE com perfil cria listing → 201
7. BUYER tenta criar listing → 403
8. Listing via ISBN preenche book do lookup (conferir book.title no banco)
9. Listing via ISBN que já existe reusa o book (não duplica)
10. Listing manual sem ISBN cria book novo
11. /books/search/fuzzy retorna resultados com score
12. Listing com 0 fotos → 400
13. Listing com priceCents < 100 → 400
14. Listing com condition inválida → 400
15. GET /listings/mine retorna listings do seller logado
16. Listing criado aparece no painel admin (M11) como PENDING_REVIEW

## Pegadinhas conhecidas

- Fuzzy match com pg_trgm: a function `similarity()` espera `text`. Se a coluna for `varchar`, pode precisar de cast.
- Auto-provision e criação de listing devem ser na MESMA transação. Se separar em dois @Transactional, seller fica órfão se listing falhar.
- Request pode vir com ISBN + manualBookData (fallback), só ISBN, ou só manualBookData. O service precisa tratar os 3.
- `priceFormatted` com locale errado: se servidor estiver em en-US, formata como "$35.00". Forçar `Locale("pt", "BR")`.
- Confirmar que `/sellers/**` e `/listings/**` são rotas autenticadas no SecurityConfig.
